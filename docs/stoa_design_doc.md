# Stoa Design Document

**Version:** 1.0.0
**Date:** 2026-03-16
**Companion to:** CAW Design Document v1.0, CAW Technical Specification v1.0.0

---

## 1. Identity

**Stoa** is the native Linux desktop GUI client for the Canonical Agent Workbench (CAW). It is a pure API client with zero embedded backend logic. Every capability surfaces through REST and WebSocket calls to a running CAW backend on localhost.

One operator. One machine. One workbench interface.

---

## 2. Purpose

CAW exposes a complete API surface (Technical Specification §13). Stoa exists to make that surface feel like a precision instrument, not a Swagger page. It translates CAW's capabilities into a coherent visual environment where the operator can chat, research, deliberate, inspect workspaces, compare models, and replay prior runs — all without leaving one application.

---

## 3. Design Principles

**3.1 Zero intelligence.** Stoa renders, routes, and displays. It does not reason, synthesize, or transform content. All cognitive work stays in CAW.

**3.2 Operator-grade density.** Information density favors power users. No gratuitous whitespace, no wizard flows, no progressive disclosure that hides controls behind clicks.

**3.3 Connection honesty.** The operator always knows the connection state, the active provider's health, and whether approval gates are pending. No silent failures.

**3.4 Approval gates are sacred.** Approval requests surface immediately, persistently, and unmissably. The operator's approval authority is never buried.

**3.5 Trace visibility.** Every run's trace is accessible without leaving the current mode. The trace panel is a global affordance, not a mode-specific feature.

**3.6 Keyboard-first.** Every primary navigation action has a keyboard shortcut. Mouse is supported; keyboard is preferred.

**3.7 Aesthetic coherence.** Dark theme. Teal accent. Monospace for data. Sans-serif for prose. Operator-grade, not consumer-grade. See §8 for tokens.

---

## 4. Operator Model

Stoa serves a single authenticated operator. Authentication uses a bearer token generated by CAW at first run, stored on disk via Tauri's IPC filesystem API with `0600` permissions. The token is never stored in `localStorage`, `sessionStorage`, or any browser-accessible storage.

On launch, Stoa reads the token from disk and attaches it to every HTTP and WebSocket request. If the token is missing or invalid, Stoa surfaces a setup screen directing the operator to the CAW key file location.

---

## 5. Modes

Stoa organizes work into six modes, each mapping to a CAW capability pillar.

**5.1 Chat.** Streaming conversational interface. WebSocket-connected. Message history sidebar. Model/provider indicator. Supports attachments.

**5.2 Research.** Source ingestion panel. Retrieval query interface. Synthesis output with inline citations linked to source excerpts. Export controls for Markdown, JSON, DOCX.

**5.3 Deliberation.** Multi-frame layout. Each frame displays its position and critique responses. Disagreement surface panel. Rhetoric analysis panel (devices, biases, inconsistencies). Configurable frame count and round count.

**5.4 Workspace.** File browser (local and remote). File viewer with syntax highlighting. Diff viewer. Patch proposal review with approve/reject controls. Command execution panel with output display. All write/execute/delete operations route through CAW's approval gates.

**5.5 Arena.** Evaluation task selector. Side-by-side run comparison. Scoring dashboard. Regression timeline.

**5.6 Replay.** Trace event timeline (filterable by event type). Event detail panel. Run summary view. Run diff view for comparing two traces.

Modes share one shell. Switching modes preserves session state.

---

## 6. Global Shell

The shell wraps all modes with three persistent elements:

**6.1 Sidebar (left).** Mode selector (six icons, Ctrl+1 through Ctrl+6). Session list for the active mode. Collapsible via Ctrl+B. Collapsed state shows icons only.

**6.2 Header (top).** Session title (editable). Connection status indicator (connected/disconnected/reconnecting). Provider health badge (healthy/degraded/down, polled per Tech Spec §4.3). Approval badge — persistent, animated when gates are pending.

**6.3 Trace panel (bottom).** Collapsible via Ctrl+T. Shows live trace events for the active session. Filterable by event type. Clicking an event opens its detail.

---

## 7. Approval Gates

Approval gates implement CAW's human-in-the-loop safety model (CAW Design Document §17). Stoa treats them as first-class UI:

- Pending count badge in header, always visible, teal-highlighted when nonzero.
- Slide-in panel from the right showing all pending requests with action description, affected resources, reversibility flag, and preview (diff or command text).
- Inline approval in Workspace mode: patch proposals and command previews include approve/deny buttons directly in context.
- Approvals are polled every 2 seconds (Tech Spec §4.3).
- Denied or timed-out requests show a dismissible notification.

## 8. Aesthetic

Dark theme only (v1.0). Background hierarchy: `#0a0e14` (base) → `#111720` (surface) → `#1a2030` (elevated). Teal accent: `#3b9eba`. Error: `#e55050`. Warning: `#d4a843`. Success: `#4caf7c`. Text primary: `#e0e4ea`. Text secondary: `#8892a0`.

Typography: IBM Plex Sans (UI text), IBM Plex Mono (code, data, traces). Base size: 14px. Scale: 12/14/16/20/24px.

Border radius: 4px (small), 6px (medium), 8px (large). Spacing unit: 4px.

All tokens are defined once in Tech Spec §10.

---

## 9. Versioning

Stoa follows Semantic Versioning 2.0.0. The version string lives in `package.json`, `tauri.conf.json`, and `Cargo.toml`. All three are updated atomically on every release. The CHANGELOG follows Keep a Changelog format. Every artifact-producing build permanently burns its version number.

---

## 10. Safety

Stoa enforces CAW's safety model at the GUI layer:

- No direct file writes, command execution, or deletion without routing through CAW's approval gates.
- The bearer token is never exposed to the WebKitGTK renderer process; all token-bearing requests route through Tauri's Rust-side IPC commands.
- No `localStorage` or `sessionStorage` usage for sensitive data.
- WebSocket reconnection uses exponential backoff with jitter; no tight retry loops.

---

## 11. What Stoa Is Not

- Not a backend. No database, no model calls, no file operations.
- Not multi-user. One operator, one token, one session at a time per mode.
- Not cross-platform (v1.0). Ubuntu 24.04 + AppImage only.
- Not a browser app. Tauri + WebKitGTK, distributed as AppImage.
